# 의도 → Figma 라이브러리 컴포넌트 빌드 4단계 검문소 워크플로우

코드/디자인 의도(또는 기존 Figma 노드)를 **Figma 디자인 시스템 라이브러리의 컴포넌트/변형세트 "정의" 자체**로 빌드·편집하는 단계형 워크플로우다.
각 단계는 **검문소(STOP)** 를 가지며, 통과 전에는 다음 단계로 넘어갈 수 없다.

> **다른 스킬과의 차이:**
> - `figma-to-code` = Figma 컴포넌트 → 코드(HTML/registry).
> - `screen-rebuild` = 레거시 화면 → V3.0 Figma **화면**(정본 컴포넌트 **인스턴스 조립**).
> - **이 스킬** = 의도/노드 → Figma **라이브러리 컴포넌트 정의**(신규 컴포넌트·**변형세트**·토큰 바인딩 저작).
> 6개 Gate를 대체하지 않는다(진행 중 사전 검문소 vs 완료 직전 사후 검문).

---

## 왜 이 스킬이 필요한가 (실패에서 역산)

라이브러리 컴포넌트 빌드/편집을 **총괄(⭐)이 혼자 빌드+검증**해서 문제가 반복됐다:
- combineAsVariants 후 패킹 누락 → 세트 폭 10100px 붕괴(렌더 떠보고서야 발견).
- 아이콘 정렬·변형세트화 누락 → 사용자가 직접 잡아줌.
근본 원인 = **빌드한 자 = 검증한 자**(자가편향) + **위임 강제 트리거 부재**(성실성 의존).

이 스킬은 그것을 **구조로 차단**한다: 빌드(🏗️)와 검증(🤖)을 분리하고, 총괄은 흐름만 관리한다.

---

## 핵심 구조 — 만드는 자 ≠ 검증하는 자, ⭐는 흐름만

| 역할 | 담당 | 단계 | 이모지 |
|------|------|------|--------|
| **원본/의도 읽기** | `figma-inspector`(기존 노드 읽을 때 실제 spawn) 또는 ⭐(의도+라이브러리 현황 파악) | 1 재고조사 | 🔍 |
| **매핑·계획** (구조·네이밍·토큰·배치 결정) | ⭐ 오케스트레이터 | 2 계획 | 🎩 |
| **빌드** | `figma-library-builder` (실제 spawn) | 3 빌드 | 🏗️ |
| **검증** | `component-verifier` (실제 spawn) | 4 검증 | 🤖 |

> **총괄(⭐)은 3·4단계를 직접 하지 않는다.** 빌드는 반드시 `figma-library-builder`에 위임하고, 검증은 반드시 별도 컨텍스트의 `component-verifier`가 한다. ⭐가 직접 use_figma로 빌드하거나 자기 결과를 "잘 됐다"고 판정하면 이 워크플로우 위반이다. (CLAUDE.md §⚖️ 운영 원칙 하드룰)
>
> **예외(⭐ 직접 허용):** 단일 노드 좌표/이름 1건 같은 **순수 기계적 미세 편집**은 ⭐가 직접 하고 렌더 1회 확인으로 갈음할 수 있다. **구조 변경(컴포넌트 생성·변형세트화·다수 노드·토큰 바인딩)은 반드시 위임**한다.

```
reports/figma-library-build/{target}/
├── 1-inventory.md     # 원본/의도 요소·기존 라이브러리 현황 (충돌·컨벤션)
├── 2-plan.md          # 빌드 계획: 컴포넌트/변형세트·variant 속성축·소스노드·네이밍·색→토큰·배치좌표 + HD
```

## 절대 규칙 (모든 단계 공통)
1. **위임 강제** — 빌드=🏗️ figma-library-builder, 검증=🤖 component-verifier. ⭐ 단독 빌드+검증 금지(구조 변경은 반드시 위임).
2. **추측 금지** — 못 읽은 값은 지어내지 말고 `get_screenshot`로 다시 읽거나 `미확인`으로 보고.
3. **색은 항상 토큰** — 새 fills는 Variable 바인딩. raw hex 잔류 = 검증 ❌.
4. **정본 우선·임의 생성 금지** — 계획에 없는 컴포넌트·variant·속성 추가 금지. 필요하면 needs-decision.
5. **원본 아이콘 강제 / 이미지 폴백** — 아이콘은 라이브러리 import. 계획이 '래스터 그대로'로 지정한 것만 이미지 보존.
6. **기존 자산 보존** — 변형세트화·리네임이 기존 인스턴스를 깨지 않게. 비운 컨테이너는 임의 삭제 말고 보고.
7. **막히면 보고** — 모호한 토큰 매핑·없는 구조·없는 아이콘은 채우지 말고 HD로 올린다.

## 1단계 — 원본/의도 재고조사 + 라이브러리 현황 (Inventory)

**담당:** 🔍 `figma-inspector`(기존 Figma 노드를 소스로 읽을 때 실제 spawn) 또는 ⭐(코드/설명이 소스일 때).

- 빌드할 대상의 **소스**(기존 노드·코드 스펙·스크린샷)를 전수로 읽어 요소·치수·색·아이콘·상태를 표로 옮긴다. (§Figma MCP 읽기 규칙: `get_metadata`로 좁히고 고른 노드만 깊이 읽기)
- **기존 라이브러리 현황**을 함께 파악한다: 같은 패밀리 컴포넌트/세트, 네이밍 컨벤션(슬래시 폴더·variant 속성 축), 배치 컬럼 좌표, 정본 토큰 컬렉션/Variable, 아이콘 라이브러리 key. → 충돌·중복·컨벤션을 1-inventory.md에 기록.
- 텍스트는 verbatim, 못 읽은 값은 이미지로 보고 채우고 출처 표기.

### 🚦 검문소 1 — 요소·현황 전수 + 사용자 확인 (STOP)
> 총 요소/대상 개수와 기존 라이브러리 충돌 여부를 명시하고 멈춘다. 확인 전 2단계로 가지 않는다.

---

## 2단계 — 매핑 & 빌드 계획 (Plan)

**담당:** 🎩 ⭐ 오케스트레이터 (구조·네이밍·토큰·배치 결정만 — 빌드는 🏗️).

각 대상을 빌드 계획으로 구체화한다:

```
| 대상 | 작업 | variant 속성축·값 | 소스 노드 | 네이밍 | 색→토큰 매핑 | 아이콘/이미지 | 배치좌표 | confidence |
|------|------|------------------|----------|--------|-------------|--------------|---------|-----------|
| StatusBar | 변형세트화 | Platform={App,Web} | 144:2129 / 155:11589 | Shell/StatusBar | (기존 바인딩 보존) | — | 9840,620 | high |
| 웹 하단내비 | 신규 컴포넌트 | — | 155:11243(래스터) | Shell/BottomNav-Web | — | 래스터 그대로 | 9840,730 | high |
```

**결정 포인트(여기서 ⭐가 정한다 — 메커니즘):** 신규 vs 변형세트화, variant 속성 축·이름, 네이밍 컨벤션, **순환참조 회피(detach 필요 여부)**, 패킹 레이아웃(패딩·갭), 배치 좌표(겹침 회피), 색→토큰 역매핑.

**필수 산출물 3가지:**
- **색상 바인딩 사전 조회표** — 소스에서 읽은 **raw hex 값을 전부** 나열하고, 각 값의 DS 토큰 등가물을 `vars-data.ts` FOUNDATION_COLOR·SEMANTIC_COLOR에서 **실제 조회**해 기록한다. 이 표 없이 검문소 2를 통과할 수 없다.
  ```
  | hex 값 | 노드·속성 유형 | DS 조회 결과 | 빌드 지시 |
  |--------|---------------|-------------|---------|
  | #ffffff | address_row fill | color/surface/default ✅ | 토큰 바인딩 필수 |
  | #353535 | nav icon stroke | color/icon/gray-dark ✅ | 토큰 바인딩 필수 |
  | #dcdcdc | tab_row fill | 없음(gray/200 근사) | → HD 등록 |
  ```
  - **등가물 있음 → 빌더에게 토큰 바인딩 필수 지시.** 허용편차로 면제 불가.
  - **등가물 없음(근사 포함) → HD로 등록, 사용자 판단** (근사 토큰 적용할지 / 원색 유지할지).
- **허용편차 선언서** — 의도된 차이를 **[노드명 + 속성 유형]** 단위로 명시. 컴포넌트 이름 수준("브라우저 크롬은 DS 색 아님")은 선언으로 인정하지 않는다.
  ```
  허용편차 #N: {노드명} 의 {속성유형} — {사유} (빌더에게: {raw 유지 / 토큰 X 사용 / 이미지 임베드})
  ```
- **결정 필요(HD) 목록** — ① 색 조회 결과 "등가물 없음" ② 정본/라이브러리에 없는 구조·세트 ③ 없는 아이콘 ④ 배치 겹침 불가피.

### 🚦 검문소 2 — 계획 1회 확인 (STOP)
> HD 목록(진짜 결정만)을 모아 사용자에게 1회 확인. 메커니즘(속성축·토큰 배선·패킹·좌표)은 ⭐가 정한다. HD 미해결이면 3단계로 안 간다. (CLAUDE.md "계획 1회 확인 후 자율 실행")

## 3단계 — 빌드 (Build)

**담당:** 🏗️ `figma-library-builder` (실제 spawn · `use_figma`)

`2-plan.md`·타깃·토큰 id·아이콘 key를 넘겨 spawn한다. 빌더는 계획대로 라이브러리 컴포넌트/변형세트를 저작하고:
- **combineAsVariants 후 variant 패킹**(좌표 재배치+세트 hug resize) — 붕괴 방지.
- **순환참조 회피**(필요 시 내부 인스턴스 detach), **토큰 바인딩**, **네이밍·배치** 규약 준수.
- figma-use 프리플라이트 엄수, 생성/변경 node id 전부 `node-map.json`에 기록.
- 빌드 직후 명백한 붕괴(패킹·치수)만 1차 자기수정. **합격 판정은 안 함** → 검증기로.

> ⭐는 빌더에게 위임만 하고 결과(node-map·needs-decision·스크린샷)를 받는다. 직접 use_figma로 빌드하지 않는다.

### 🚦 검문소 3 — needs-decision 0 (STOP)
> 빌더가 올린 needs-decision/비운 컨테이너가 있으면 2단계로 되돌리거나 사용자 확인. 없으면 4단계.

---

## 4단계 — 검증 (Verify)

**담당:** 🤖 `component-verifier` (실제 spawn, 빌더와 분리)

`2-plan.md`·원본 스크린샷·`node-map.json`을 기준으로 빌드 결과(Figma 노드)를 대조한다(에이전트 정의 §(C) Figma 라이브러리 빌드 검증):
- **기계:** variant 전수·속성, **패킹 정상(붕괴 ❌)**, 토큰 바인딩, 순환참조 0, 네이밍, 기존 인스턴스 무결성, 원본 아이콘/이미지.
- **렌더:** 각 variant `get_screenshot` 시각 대조.
- **두갈래 분류로 반환:** ❌(a) 코드/빌드 실수 → 수정 / 🟡(b) 허용편차 → 유지 / ❓(c) 애매 → 사용자 확인.

결과를 `4-verification.md`에 저장.

### 🚦 검문소 4 — ❌(a) 0 · ❓(c) 0 (반복 STOP)
> ❌(a)가 0이 될 때까지 3단계로 되돌려 반복. ❓(c) 있으면 사용자 확인 전 완료 보고 금지. 🟡(b)는 선언서에 남기면 통과.

## 테스트 시나리오

- **정상:** "이 웹 상태바·하단 내비바를 Shell 세트에 추가해줘" → 1단계 figma-inspector가 원본+Shell 현황 전수 → 2단계 ⭐가 변형세트화/신규 계획+HD 확인 → 3단계 figma-library-builder가 combineAsVariants+패킹+토큰 → 4단계 component-verifier가 패킹 붕괴·토큰·렌더 대조 ❌0 → Summary(🔍🎩🏗️🤖 모두 표시).
- **에러:** 빌더가 패킹 누락으로 세트 폭 붕괴 → 검증기가 "variant 패킹 ❌(비정상 bounds)"로 적발 → 3단계 반환·재패킹 후 재검증. (⭐가 자가통과시키지 않음)
