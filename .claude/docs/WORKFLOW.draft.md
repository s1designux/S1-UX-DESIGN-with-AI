## 0. 운영 모델 (✅ 확정 · 논의 끝)

복제-배포 모델. 라이브러리 구독 아님.
각 프로젝트 Figma 폴더에 컴포넌트 세트+배리어블을 깔고 **그 안에서만** 작업한다.

- 전파 없음이 곧 안전장치다. 한 프로젝트를 고쳐도 다른 프로젝트는 미동 없음.
- 원본 파일에서 재설치해도 이미 배포된 프로젝트엔 영향 없음.
- **라이브러리 승격은 고려 대상 아님.** 전파가 오히려 위험해서 일부러 격리하는 구조다.

## 1. 파이프라인 (✅ 확정 · 기계적)

```
vars-data.ts (단일 정본)
   ├→ gen-semantic-tokens.js → tokens.css        (웹)
   └→ build-components.ts    → Figma 컴포넌트 빌드 (Figma)
```

- 데이터 흐름은 **코드 → Figma 단방향.**
- 토큰은 **반드시 vars-data.ts 에 정의** → gen 스크립트가 tokens.css 생성.
  tokens.css 를 직접 편집하는 것은 우회(bypass)이며 사고다.
- 배포물 zip(`assets/downloads/s1-ux-design-guide-installer.zip`)은
  디자이너가 웹(install-prompt.html)에서 내려받는 파일이며 **git 추적 대상.**
  소스 변경 시 리빌드해 같은 커밋에 동봉한다.

> 💡 **주의(✅ ②③ 실측):** "tokens.css 에 값이 있다" ≠ "우회 배선이다".
> gen 스크립트가 만든 자동생성 블록(GEN:SEMANTIC 등, "수동 편집 금지" 주석 안)이면 **정상**이다.
> 작업 ③ 에서 chip 을 "우회"로 오판할 뻔했으나, 조사 단계가 이를 바로잡았다.
> → **핸드오프 전제를 믿지 말고, 조사가 전제를 검증하게 하라.** (아래 STEP 3-1)

## 2. STEP 1 — 작업 루트 판단 (매번 첫 질문)

> **"이번 일은 A~E 중 무엇인가?"** 루트가 정해지면 손댈 파일·순서·건드리는 표면이 따라온다.

| 루트 | 무슨 일 | 검증 사례 | 토큰 |
|---|---|---|---|
| **A. 신규 편입** | 레거시/시안/서비스패턴 → 컴포넌트 변환 → 등록 | 🧪 미실행 | 대개 신설·정리 |
| **B. 상태·배리언트 추가** | 등록된 것에 상태 추가 | ✅ date-picker Hover(②) | 기존 배선 or 신설 |
| **C. 배선 정정** | 잘못 연결된 토큰 바로잡기 | ✅ form-control 캐럿(1f9444c) | 정정만 |
| **D. 배리언트 삭제** | 불필요 상태 제거 | ✅ dropdown Disabled(4e1f85f) | 대개 안 건드림 |
| **E. 토큰 (신설·정정)** | 컴포넌트 늘며 새 토큰 필요 | ✅ selected-hover 신설(③) | 신설이 주작업 |

> 🧪 **루트 A(신규 편입)는 아직 안 해봤다.** 처음 할 때 아래를 특히 주의:
> - 밖에서 온 것은 십중팔구 토큰이 안 맞는다(레거시 옛 토큰·시안 하드코딩 hex·없는 토큰 요구).
> - core/service 소속을 편입 시점에 정해야 한다(섞이면 나중에 못 푼다).
> - 기존 컴포넌트와 이름·역할 충돌 여부 확인(dep-set detach 위험).
> → 실행 후 이 항목을 ✅ 로 승격 검토.

## 3. STEP 2 — 여러 개 동시 진행 시 (✅ ②③ 부분 검증)

원칙: **"따로 만들고, 함께 싣는다."**

1. **조사는 한꺼번에 OK** — 서로 얽히는지·같은 토큰 쓰는지 먼저 봐야 함.
2. **수정·검증은 하나씩 순차** — 하나라도 verifier FAIL 이면 거기서 정지. 다음으로 안 넘어감.
3. **커밋 분리 vs 묶음 기준: "이 중 하나만 나중에 되돌릴 일이 있는가?"**
   - 서로 다른 컴포넌트 동시 편입 → **커밋 분리** (하나만 되돌리기 위해)
   - 동일 유형 일괄 정정 → **한 커밋 OK** (한 덩어리. 메시지에 대상 명시)
4. **리빌드는 맨 마지막 1회** — 매번 리빌드 = 낭비 + stash 절차 반복.

> ✅ 작업 ② 에서 date-picker 코드와 BACKLOG 를 각각 커밋하고 리빌드는 코드 커밋에만 붙인 것이 이 원리의 실제 적용.

## 5. ★ 동기화 체크 (✅ 확정 · 이번 논의의 알맹이 · 출구에서 반드시)

**문제의 뿌리:** 하나의 컴포넌트가 여러 표면에 흔적을 남기는데 자동 동기화가 안 된다.
지금까지 사고 대부분(스테일 주석·tokens.css 우회·chip 타임밤·낡은 zip)이 여기서 발생했다.

### 두 층위로 나눠 본다

- **층위 1 — 값 일치:** 같은 토큰이 표면마다 같은 값인가.
  기계 대조 가능. **소비 화면을 '나열'하지 말 것** — 화면은 계속 늘어 목록은 반드시 뒤처진다.
  원리로: *"토큰 쓰는 모든 화면은 tokens.css 정본을 봐야 한다. 하드코딩 hex = 드리프트."*
  → 나열이 아닌 **스캔**으로 잡는다 (scan-system-map.js 계열 활용 여지).
- **층위 2 — 배선:** 그 토큰을 올바른 요소에 썼나.
  **판단 필요 → 자동화 불가.** 자동화하면 date-picker hover 버그 같은 걸 양산한다.
  여기는 **검증(육안+게이트)** 이 답이지 자동화가 아니다.

### 출구 체크리스트 (루트마다 걸리는 항목이 다름)

- [ ] **tokens.css** — gen 재실행했나 (직접 편집 아님)
- [ ] **Figma / zip** — 리빌드했나 (아래 Gate 6 주의)
- [ ] **웹·문서 화면** — 정본 배선 + 하드코딩 hex 없음
      (components-new.html · semantic.html · install-prompt.html · foundation.html · token download 등 **소비 화면 전부**)
- [ ] **registry** — ⚠️ 미검증 baseline. 지금은 수동 갱신 or "의도적 보류" 명시. (§7 참조)
- [ ] **주석 / 문서** — 코드와 불일치 없음 (스테일 주석 재발 방지)
- [ ] **육안 확인** — 자동 검증이 못 잡는 상태(예: :hover)는 브라우저에서 직접 볼 것

> ✅ **작업 ③ 실측 — selected-hover 토큰 1개 신설 시 전파된 표면:**
> | 표면 | 무엇을 | 방식 |
> |---|---|---|
> | vars-data.ts | 토큰 신설(정본) | 수동 |
> | tokens.css | 재생성 | gen (직접편집 금지) |
> | semantic.html | 토큰 표 행 추가 | 수동 |
> | install-prompt.html | 인라인 CSS(light·dark) | 수동 |
> | components-new.html | 웹 CSS 규칙 | 수동 |
> | build-components.ts | Figma 배리언트 | (Figma 건드릴 때만) |
> | 설치기 zip | 리빌드 | (Figma·신규토큰일 때) |
> 자동 3 · 수동 3 · 조건부 2. **웹 하나만 고치고 끝내면 나머지가 드리프트한다.**

### ⚠️ Gate 6(installer freshness) 사각지대 (✅ ②③ 확인)

Gate 6 은 zip 안 code.js 에 **토큰 키가 embed 됐는지만** 본다.

- **새 토큰 신설** → 키가 zip 에 없으니 게이트가 운다 → 리빌드 강제됨 (작업 ③).
- **기존 토큰 배선만** (구조·배리언트만 변경) → 키는 이미 있어 게이트가 **통과시킨다** → 리빌드를 강제 못 함 (작업 ②).
  → 이 경우 리빌드를 안 하면 배포 zip 에 변경이 빠진 채 남는다.
  **게이트만 믿지 말고, 구조가 바뀌었으면 리빌드 필요 여부를 직접 판단하라.**

## 6. 파이프라인 실행 순서 (✅ ②③ 검증 · 승인 후에만)

1. vars-data.ts 정본 수정/신설
2. gen 스크립트 재실행 → tokens.css 재생성 (**직접 편집 금지**)
3. 웹·문서 소비 표면 배선 수정
4. (필요 시) build-components.ts 배리언트 수정
5. **diff 검증** — 의도한 부분만 변했는지. 범위 밖이 변했으면 즉시 정지·보고.
6. 빌더 재실행 → 예외 없이 완주하는지
7. **component-verifier 정식 spawn 검증** (자가인증·`--change mechanical` 경로 금지).
   Figma 컴포넌트 변경이 0 이면 "verifier 대상 없음"을 보고하고 넘어감 (작업 ③ 사례).
8. **gate:check** — 게이트가 울면 내용 그대로 보고. `--no-verify`·`--update-baseline` 금지.
9. (Figma·신규토큰 변경 시) **리빌드** — §8 stash 절차 준수.
10. **정지. 스테이징 목록만 제안.** 직접 add·커밋·push 금지.

## 7. registry 정상화 — 별도 트랙 (🧪 미실행 · 작업 루트와 분리)

**걱정의 핵심:** 검증 안 된 registry 위에 자동 생성기를 얹으면
"틀린 베이스를 자동으로 대량 재생산"한다. = 썩은 토대 위 자동화.

**원칙:** 자동화·빌드를 얹기 전에 그 아래 베이스가 검증돼 있어야 한다.

```
1. registry 의 '올바른 형태' 정의 (무슨 필드·무슨 근거)
2. 현재 registry 를 그 형태로 한 번 바로잡기 (수동·검증)
3. 바로잡힌 상태를 baseline 으로 고정
4. ── 이 지점 통과 후에만 ──
5. 자동 생성기 구축 (검증된 baseline 재생산)
```

> 현재 1번도 미완. **당장 자동화 금지.** '미루는' 게 아니라 '토대 먼저'.
> 그때까지 registry 는 "미갱신 상태 · 근거로 쓸 수 없음"으로 취급. 수동 갱신 or 보류 명시.

## 8. 리빌드 stash 절차 (✅ ②③ 검증)

GUI 검수기 미커밋 변경(code.ts·ui.html)이 있으면 리빌드가 배포물을 오염시킨다.
리빌드 전에 그 두 파일만 경로 지정 stash 로 대피.

```
0. 사전 기록 — 스테이징 목록 + 대상 파일 blob 해시·크기
1. git stash push -- <code.ts 경로> <ui.html 경로>   (경로 지정. --keep-index 금지)
2. 리빌드
3. git stash pop
4. blob 해시·크기가 0단계 기준값과 완전히 일치하는지 대조. 어긋나면 정지·보고.
5. (pop 후 unstaged 로 풀림) → 다시 git add 는 river 가 직접.
```

- **`--keep-index` 절대 금지** (과거 사고 05fce64 원인). stash 를 먼저, 스테이징은 나중에.
- **git stash drop 은 river 승인 후에만.**
- 다중 세션이 같은 작업트리를 공유하면, **리빌드·커밋·stash 구간에 다른 세션이 인덱스를 건드리지 않도록** 조율. (git 인덱스와 zip 은 저장소에 하나뿐이다.)

## 9. 작업 원칙 (✅ 확정 · 엄수)

- **git add . 금지.** 선택 스테이징만.
- **커밋 전 반드시 정지하고 보고.** 자체 판단 커밋 금지.
- **자기 검증 금지.** component-verifier 정식 spawn 으로 검증. orchestrator 자가인증 경로 금지.
- **`--no-verify` 절대 금지.** 게이트가 울면 내용 그대로 보고.
- **`--update-baseline` 등 임의 승인 금지.**
- **push 하지 말 것.**
- 설계 판단이 필요하면 멈추고 물을 것.
- 코드와 주석이 어긋나지 않게 할 것.
