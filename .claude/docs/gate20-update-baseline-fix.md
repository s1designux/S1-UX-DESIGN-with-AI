# Gate 20 `--update-baseline` 개선 설계 (A안: 줄이기 전용)

> 상태: **설계 확정 · 구현 대기.** 이 문서는 이후 구현의 기준이며, 다른 세션과 방향을 공유하는 근거다.
> 작성 시점: 2026-07-15 · 조사 세션 기준 · 코드는 아직 손대지 않았다.
> 대상 코드: `scripts/token-drift-check.js` (Gate 20 · 토큰 drift 검사기) — **본 문서 작성 시점엔 미수정.**

---

## 1. 문제 요약 (쉬운 말)

Gate 20은 registry의 설명용 데이터가 **"최신 컴포넌트가 안 쓰는 낡은 토큰"**을 언급하면 잡아내는 검사기다.
아직 못 고친 낡은 언급은 `registry-drift-baseline.json`이라는 **"묵인 목록(=이미 아는 backlog)"**에 적어 두고,
**새로** 생긴 낡은 언급만 골라 커밋을 차단한다(래칫 = 한 방향 잠금).

문제는 이 묵인 목록을 갱신하는 `--update-baseline` 옵션이다.

- **본래 역할(기대):** 청소기 — 이미 *고쳐서 사라진* 항목을 목록에서 지워 목록을 줄여준다.
- **실제 동작(버그):** 복사기 — **지금 이 순간 감지된 낡은 언급 전부를 그대로 통째로 "정답"으로 복사해 박제**한다.
  줄이는지 늘리는지 구분하지 않고, 현재 상태를 무조건 승인한다.

### 취약 지점 (정확한 위치)

**`scripts/token-drift-check.js:115-122`** (`if (UPDATE) { ... }` 블록)

```js
115  if (UPDATE) {
116    fs.writeFileSync(BASELINE, JSON.stringify({
117      note: '...',
118      count: currentKeys.size, knownDrift: [...currentKeys].sort(),   // ← 현재 drift 전부를 통째로 기록
119    }, null, 2) + '\n');
```

- 113줄에서 만든 `currentKeys`(지금 감지된 drift 전부)를 118줄에서 **그대로** baseline에 쓴다.
- 113줄과 116줄 사이에 **"이게 진짜 아는 backlog인가, 방금 새로 생긴 오류인가"를 가르는 판단이 없다.**
  추가되는 항목을 막는 가드가 전혀 없다.
- tokens.css 대조는 drift를 *판별*할 때만 쓰이고(78~112줄), update 경로는 그 판별 결과를 **무비판 승인**한다.

### 왜 위험한가

작업 중 **반쯤 고친 틀린 상태**(정본도 옛것도 아닌 중간값)가 워크트리에 있을 때 `--update-baseline`을 돌리면,
그 **틀린 상태가 그대로 "정답 baseline"으로 박제**된다.
이후 Gate 20은 그 틀린 상태를 **다시는 지적하지 않는다.** → "틀린 걸 정답으로 굳힘"(BACKLOG #2 경고 그대로).

## 2. 결정된 개선 방향 — A안 (줄이기 전용)

`--update-baseline`을 **한 방향(줄이기)으로만** 작동하게 만든다.

### 규칙

1. **해소분(baseline에 있었는데 지금 사라진 항목) 제거만 허용.** → 목록이 줄어든다(정상).
2. **새 항목(지금 감지됐는데 baseline엔 없던 것) 추가는 거부.**
3. 새 항목이 감지되면:
   - **에러로 차단**(baseline 파일을 쓰지 않고 종료, 비정상 종료코드).
   - **추가되려던 N건 목록을 그대로 출력**한다.
   - **"이 항목들은 baseline에 넣지 말고 먼저 고치세요"** 라고 안내한다.

즉 `--update-baseline`은 이제 **"청소 전용"**이다. 늘리는 길이 코드에 아예 없다.

### B안·C안을 지금 안 쓰는 근거

- **B안(사람 확인 프롬프트)** — 결국 그 순간 사람 판단에 의존하며, 공유 워크트리·자동화에선 무심코 통과된다. 기계 차단(A안)보다 약하다.
- **C안(사유 큐레이션 목록)** — 가장 튼튼하나 기존 86건에 사유를 달아 이관하는 초기 비용이 크다. 지금 #1 착수를 막을 만큼 급하지 않다.
- **C안은 향후 별도 과제로 남긴다.** (A안이 "새 오류 세탁"이라는 급한 구멍을 먼저 막고, 감사성 강화는 나중에.)

## 3. 구현 시 체크리스트 (다음에 코드 짤 때 볼 것)

- [ ] **줄이기 vs 늘리기 구분:** update 경로에서 `currentKeys`를 통째로 쓰지 말고,
      기존 baseline과 비교해 `added = currentKeys − baseline` / `removed = baseline − currentKeys`를 계산한다.
      - `added.length > 0` → **차단**(파일 미기록 + 비정상 종료).
      - `added.length === 0` → 새 baseline = `currentKeys`(= 기존에서 removed만 빠진 상태). 정상 기록.
- [ ] **친절한 에러 메시지:** 차단 시 추가되려던 항목 N건을 `drift.get(key)` 설명과 함께 나열하고,
      "baseline에 넣지 말고 먼저 정본 토큰으로 고치라"는 안내를 붙인다. (평소 검사의 ❌ 출력과 톤 일치)
- [ ] **정상 흐름 재확인:** 별칭을 *올바르게* 고치면 그 컴포넌트가 정답 토큰을 쓰게 되어 drift가 **줄고**,
      그때 `--update-baseline`이 removed만 반영해 baseline을 **축소**한다. 이 경로만 열려 있어야 한다.
- [ ] **baseline 자체는 이 작업에서 손대지 않는다**(값 갱신은 구현·검증 후 별도 판단).
- [ ] 코드 수정이므로 ⭐ 단독 자가검증 금지 — 실제 렌더 대상은 아니나, 게이트 로직 변경은 적대적 테스트
      (새 stale 1건 주입 → 차단되는지 / 해소분만 있을 때 정상 축소되는지)로 확인한다.

## 4. BACKLOG 연결

- 이 개선은 **BACKLOG #1(별칭 철거 마무리 · 8개 컴포넌트)의 선행 안전장치**다.
  #1은 registry의 낡은 별칭을 정본 토큰으로 바꾸는 작업이라 Gate 20이 대량 반응하는데,
  현재 `--update-baseline`은 그 와중의 틀린 중간 상태를 굳힐 수 있다(BACKLOG #2).
- **명문 규칙: #1 별칭 철거 작업 중에는 `--update-baseline`을 절대 실행하지 않는다.**
  (A안 구현 전까지는 도구 자체가 위험하고, 구현 후에도 "고친 뒤 축소"만 정상 경로다.)
- 관련: BACKLOG #2(Gate 20 `--update-baseline` 취약점) = 본 문서가 다루는 대상.
