Harness Overview

하네스가 무엇인지, 왜 만들고 있는지, 어떤 식으로 동작하는지 — 비개발자를 위한 설명서

Harness Admin
🧭
Harness란?
디자인시스템을 일관되게 운영하기 위한
자동화 운영 환경입니다
"하네스(Harness)"는 말 그대로 여러 부품을 하나로 묶어 제어하는 장치입니다. 여기서는 Figma · CSS 토큰 · AI · 서비스 코드를 하나의 규칙 아래 연결하고, 불일치를 자동으로 감지·기록·수정하는 디자인시스템 운영 환경을 의미합니다.

토큰 검증 가이드 자동화 Figma 동기화 Source Guard AI 작업 기준 이 다섯 가지 기능이 하나의 운영 환경으로 통합되어 있습니다.
왜 만들었나 — 우리가 겪던 문제들
PROBLEM 01
Figma 동기화 토큰 검증
Figma Variables와 CSS 체계가 달라
자동 싱크가 불가능했다
Figma는 디자인 변수를 color/text/primary 형식으로 관리하고, CSS는 --color-text-primary 형식으로 관리합니다. 같은 설계 의도를 담고 있어도 두 체계를 자동으로 연결하는 구조가 없었기 때문에, Figma에서 값이 바뀌어도 코드에 반영하려면 사람이 직접 찾아 번역해야 했습니다. 그 사이 간격에서 불일치가 쌓입니다.
예: Figma에서 color/action/primary/default 값을 수정해도, CSS의 --color-action-primary-default는 자동으로 바뀌지 않아 개발자가 수동으로 파악·반영하기 전까지 화면과 디자인이 어긋남
PROBLEM 02
Source Guard
개발 결과물에 디자인시스템이
제대로 반영되지 않았다
토큰 이름 대신 HEX 코드를 직접 쓰거나, 공식 변수 대신 하드코딩된 값이 들어가는 일이 반복됐습니다. 검사 기준이 없으면 규칙을 어겨도 아무도 모릅니다. 코드 리뷰에서 일일이 찾아내는 것도 한계가 있습니다.
예: color: #0072CE 가 수십 개 파일에 흩어져 있어도 자동으로 잡아내는 도구가 없었음
PROBLEM 03
AI 작업 기준 가이드 자동화
기준이 없으면 AI도, 신규 팀원도
임의로 작업하게 된다
공식 토큰 구조와 네이밍 규칙이 문서화되지 않으면, AI 도구나 새로 합류한 팀원이 기존 체계를 모른 채 임의로 값을 만들거나 잘못된 토큰명을 사용합니다. 실수가 개별적으로 쌓이면 시스템 전체가 조금씩 어긋납니다.
예: AI(Claude)가 Figma 원본을 확인하지 않고 토큰 이름을 추측해 생성하면, 실제 Registry와 충돌하는 값이 코드에 들어감
지금 무엇을 만들고 있나?
세 가지 문제를 동시에 해결하는 자동화 구조
① 토큰 구조 정리
Figma Variables와 CSS 토큰을 Foundation → Semantic → Component 3단계로 정리합니다. 디자이너와 개발자가 같은 이름으로 소통할 수 있게 됩니다.
② 정합성 자동 검사
실제 서비스 코드에서 HEX 직접 사용·규칙 위반을 자동으로 찾아냅니다. 사람이 일일이 확인하지 않아도 됩니다.
③ AI 작업 기준 제공
Claude가 이 가이드를 읽고 토큰 이름·값·구조를 스스로 참고하며 작업합니다. 임의로 값을 만들지 않고 항상 Registry를 먼저 확인합니다.

Harness Automation Flow

색상 토큰 검수 흐름 예시 — border-color: #0072CE 가 발견됐을 때 무슨 일이 일어나는가
STEP 1
Legacy Audit
과거 기준·비표준 사용 정리
지금까지 만들어진 화면에서 규칙에 어긋나는 코드가 있는지 먼저 훑어봅니다.
기존 화면이나 CSS에서
border-color: #0072CE;
처럼 HEX 직접 사용 사례 확인
↓
STEP 2
Canonical Registry
최신 공식 기준 확인
어떤 이름이 공식 토큰인지, 어떤 값을 써야 하는지 — 정답지를 불러옵니다.
Input focus border의 공식 기준
--input-focus-border
semantic 기준: --color-form-control-border-focus
↓
STEP 3
Token Mapping
Figma ↔ CSS 토큰 연결
Figma 변수 이름과 CSS 토큰 이름을 연결합니다. 디자이너 언어와 개발자 언어를 번역하는 단계입니다.
Figma Input / Focus / Border
↔ var(--input-focus-border)
↔ --color-form-control-border-focus
↓
STEP 4
Token Audit
실제 작업물 정합성 검사
실제 코드에서 공식 토큰 대신 색상 값을 직접 입력한 곳이 있는지 자동으로 찾아냅니다.
CSS/HTML에서
border-color: #0072CE; 발견
→ 공식 토큰 미사용으로 감지
↓
STEP 5
Source Guard
규칙 위반 판단
찾아낸 항목이 실제 위반인지 판단하고 심각도를 분류합니다. 예외로 허용된 경우는 통과시킵니다.
HEX 직접 사용 금지
규칙 위반으로 분류
↓
STEP 6
Fix Candidate
수정 후보 생성
위반된 코드를 어떻게 고치면 되는지 제안합니다. 사람이 직접 찾아서 수정하지 않아도 됩니다.
border-color: #0072CE; ↓ border-color: var(--input-focus-border);
↓
STEP 7
Reports / CI
결과 기록 및 자동화 연결
검사 결과를 보고서로 저장하고, 매일 자동으로 반복 실행되도록 설정합니다.
hardcoded color 1건
fix candidate 1건
review needed 0건
하네스를 구성하는 것들
🎨
토큰 (Token)
색상·간격·크기에 이름을 붙인 것입니다. --color-blue-300 처럼요. HEX 값 대신 이 이름을 쓰면 테마를 바꿔도 자동으로 반영됩니다.
📋
Registry (레지스트리)
토큰의 공식 목록표입니다. 어떤 토큰이 존재하는지, 어떤 값을 가지는지, Figma의 어떤 변수와 연결되는지 기록되어 있습니다.
🛡️
Source Guard
실제 서비스 코드를 스캔해서 규칙 위반(HEX 직접 사용, ghost 버튼 등)을 찾아내는 검사기입니다. 사람이 일일이 확인하지 않아도 됩니다.
🤖
AI (Claude)
이 가이드를 읽고 토큰 구조와 규칙을 이해한 뒤 작업합니다. 임의로 값을 만들지 않고, 항상 Registry를 먼저 확인합니다.
🔗
Figma 연동
Figma의 Variables(변수)와 CSS 토큰을 의미 기반으로 매핑합니다. 디자이너가 Figma에서 수정하면 코드에도 동일한 기준이 적용됩니다.
📊
Reports
모든 검수·의사결정·수정 내역을 문서로 기록합니다. 나중에 "왜 이렇게 됐지?" 라는 질문에 답할 수 있는 근거가 됩니다.
자주 나오는 용어
용어 설명 예시
Foundation Token 가장 기본이 되는 값. 색상 팔레트처럼 원재료입니다. --color-blue-300: #0072CE
Semantic Token 역할을 이름으로 표현한 토큰. "이 색은 액션 버튼에 쓰는 색"처럼요. --color-action-primary-default
Component Token 특정 컴포넌트에서만 쓰는 토큰. Button, Input 각자의 별칭입니다. --button-primary-default-bg
HEX 직접 사용 토큰 이름 대신 색상 코드를 그대로 쓰는 것. 규칙 위반입니다. color: #0072CE ← 금지
Figma Variables Figma 안에서 정의된 디자인 토큰. CSS 토큰과 매핑됩니다. Figma Input/Focus/Border
Human Decision (HD) AI가 임의로 결정하지 않고 사람의 확인이 필요한 항목입니다. HD-4: correct vs success 명칭 확정
Canonical 여러 이름·값 중 공식으로 확정된 기준입니다. correct가 canonical, success는 Figma alias
