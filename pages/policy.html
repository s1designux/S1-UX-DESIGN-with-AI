1. 토큰 4계층 구조
Primitive → Semantic → Component → Legacy 계층별 역할과 참조 규칙
💡
핵심 원칙
상위 계층은 반드시 하위 계층을 var()로 참조합니다. 계층을 건너뛰거나, 상위에서 하위를 직접 참조하는 것은 금지입니다.
Primitive
원천 색상값. hex 직접 정의 허용.
--color-blue-300: #2b9eff
hex 정의
↓
Semantic
역할 기반 토큰. Primitive를 var()로 참조. Light/Dark 분리.
--color-action-primary: var(--color-blue-300)
var() 참조
↓
Component
컴포넌트 전용 토큰. Semantic을 var()로 참조.
--button-primary-bg: var(--color-action-primary)
var() 참조
↓
Legacy
구버전 토큰. 신규 Semantic으로 포워딩.
--legacy-text-primary: var(--color-text-primary)
포워딩
CSS — 올바른 참조 예시 
/* ✅ Primitive 정의: hex 직접 사용 허용 */
--color-blue-300: #2b9eff;

/* ✅ Semantic 정의: Primitive를 var()로 참조 */
--color-action-primary: var(--color-blue-300);

/* ✅ Component 정의: Semantic을 var()로 참조 */
--button-primary-bg: var(--color-action-primary);

/* ❌ 금지: Component에서 Primitive 직접 참조 */
--button-primary-bg: var(--color-blue-300);

/* ❌ 금지: Semantic에서 hex 직접 사용 */
--color-action-primary: #2b9eff;

2. 신규 vs 레거시 사용 기준
어떤 화면·컴포넌트에 신규 토큰을 써야 하는지, 레거시 허용 범위는 어디까지인지
구분 신규 토큰 사용 Legacy 허용 비고
신규 화면 개발 ✓ 필수 ✗ 금지 신규 기능은 반드시 신규 토큰 사용
기존 화면 대규모 개편 ✓ 필수 ✗ 금지 리뉴얼 범위 내 전체 마이그레이션
기존 화면 소규모 수정 △ 권장 △ 조건부 허용 수정 영역만이라도 신규 토큰으로 교체 권장
버그 수정 (UI 변경 없음) △ 선택 ✓ 허용 기능 수정 목적, 불필요한 토큰 변경 자제
레거시 유지보수 전용 화면 △ 선택 ✓ 허용 레거시 화면 포워딩 토큰 참조 허용
⚠️
Legacy 토큰 사용 시 주의사항
Legacy 토큰(--legacy-*)은 신규 Semantic 토큰으로 포워딩되어 있습니다. 직접 hex를 참조하는 구버전 변수는 포워딩 대상에서 제외될 수 있으므로, 마이그레이션 가이드를 반드시 확인하세요.

3. 토큰 명명 규칙
CSS Custom Property 이름 작성 원칙
1
kebab-case + 소문자 필수
모든 토큰명은 -- 접두사 + kebab-case(소문자, 하이픈 구분). 대문자, 언더스코어, camelCase 금지.
--color-action-primary ✓   --colorActionPrimary ✗   --Color_Action ✗
2
카테고리 / 역할 / 상태 순서
패턴: --{카테고리}-{역할}-{상태(선택)}
예: --color-action-primary, --button-primary-hover-bg, --color-text-disabled
3
Figma 변수명 변환 규칙
Figma 변수명 → CSS 변수명 변환 시: / → -, 공백 → -, & 제거, 대문자 → 소문자.
color/action/primary → --color-action-primary
4
Layer 접두사 고정
Primitive: --color-{palette}-{step} (예: --color-blue-300)
Semantic: --color-{role}-{variant} (예: --color-bg-default)
Component: --{component}-{property} (예: --button-primary-bg)
Legacy: --legacy-{originalName} (예: --legacy-text-primary)
🌗
4. 다크모드 대응 정책
Light/Dark 테마 전환 구조 및 구현 방식
ℹ️
현재 상태
다크모드 토큰 구조는 정의되어 있으나, 실제 다크 값은 일부 placeholder 상태입니다. Semantic Tokens 페이지 완성 후 전체 다크 값이 확정됩니다.
CSS — 테마 구조 
/* Light (기본값): :root에 정의 */
:root {
  --color-bg-default: #ffffff;
  --color-text-primary: #000000;
  --color-action-primary: #0072ce;
}

/* Dark: [data-theme="dark"] 속성으로 오버라이드 */
[data-theme="dark"] {
  --color-bg-default: #1a1a1a;
  --color-text-primary: #f0f0f0;
  --color-action-primary: #2b9eff;
}
JS — 테마 전환 
// document.documentElement에 data-theme 속성 토글
document.documentElement.setAttribute('data-theme', 'dark');
document.documentElement.removeAttribute('data-theme'); // light 복원

// Tailwind 사용 시: tailwind.config.js에 darkMode 설정 필요
// darkMode: ['selector', '[data-theme="dark"]']
✅ 다크모드 대응 권장
Semantic token(--color-bg-default)만 사용 — 자동 전환
SVG 아이콘에 fill: currentColor 사용
이미지는 filter: brightness()로 보정 고려
Gradient는 시작·끝 모두 Semantic token 사용
다크 값 확정 전 디자이너와 협의 후 개발
❌ 금지 사항
Primitive token 직접 사용 — 테마 전환 불가
hex 하드코딩 — 다크 전환 안됨
prefers-color-scheme만 사용 — data-theme과 충돌
SVG 아이콘에 fill="#000" 고정
!important로 테마 오버라이드 차단

5. 색상 사용 원칙
색상 선택 기준, 접근성 요건, CI 색상 사용 제한
1
접근성 — WCAG AA 필수
모든 텍스트-배경 조합은 명도 대비 4.5:1 이상(AA 기준)을 충족해야 합니다. 장식 요소(아이콘, 구분선)는 3:1 이상. 대형 텍스트(18px Bold 이상)는 3:1 이상.
2
CI / Logo 색상 제한
--color-brand-ci (#004097)는 로고·CI 전용입니다. 일반 UI 버튼, 링크, 배경에 직접 사용 금지. CTA에는 --color-brand-blue (#0072ce) 또는 --color-action-primary를 사용하세요.
3
상태 색상 일관성
에러·위험: --color-brand-red / --color-action-danger 계열
성공·확인: Green 팔레트 (--color-green-300~400 기준)
경고: Orange / Yellow 팔레트
정보: Blue 팔레트
4
데이터 시각화 팔레트 분리 (예정)
차트·그래프에서는 UI 팔레트와 별도의 Data Visualization 팔레트를 사용합니다. (Semantic Tokens 2단계에서 정의 예정)

6. 컴포넌트 토큰 정책
컴포넌트 레벨 토큰의 정의 기준과 스코프
항목 기준
등록 조건 3개 이상 화면에서 동일 패턴으로 반복되고, Semantic token 조합만으로 표현 불가한 경우
참조 허용 Semantic token만 var()로 참조. Primitive 직접 참조 금지
스코프 :root 전역 정의. 컴포넌트 내부에서만 사용하고 외부에서 직접 참조 금지
다크 대응 Semantic을 var()로 참조하면 자동 대응. 별도 다크 오버라이드 최소화
네이밍 --{component}-{property}-{state} 패턴. 예: --button-primary-hover-bg
⚠️
Component 토큰 남용 주의
모든 값을 Component 토큰으로 만들면 오히려 복잡도가 증가합니다. Semantic token으로 직접 표현 가능한 경우 Component 토큰을 별도로 만들지 마세요.
