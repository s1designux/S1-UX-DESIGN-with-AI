👥 에이전트 팀 구성

검사기(🔎)가 '정해둔 규칙대로인가'를 기계적으로 자동 점검한다면, 여기 팀원들은 읽고·만들고·판단하는 일을 합니다 — 규칙으로 딱 떨어지지 않아 사람 같은 판단이 필요한 일이라 검사기가 아니라 에이전트입니다. 필요할 때만 총괄이 불러서 그 일만 맡기고, 끝나면 결과를 보고합니다. 특히 만드는 일과 검증하는 일은 일부러 다른 팀원이 맡습니다(자기 일을 자기가 검사하면 관대해지니까).

⭐ 총괄 총괄
river님의 목표를 받아 계획을 세우고, 아래 도우미들을 부르고, 검사기를 돌려 결과를 책임집니다.
🤖 Figma원본 조사
피그마 원본의 크기·색·구조를 직접 읽어옵니다. 무엇이 진짜 의도인지 판단하며 읽는 일이라 검사기로는 못 합니다.
1단계 · 원본 읽기
🤖 가이드화면 제작
읽어온 원본으로 홈페이지 가이드 화면(HTML)을 새로 짓거나 고칩니다. 없던 화면을 '창작'하는 일입니다.
3단계 · 만들기
🤖 원본대조 검증
만든 결과를 원본과 한 항목씩 맞대어 틀린 곳을 찾습니다. 만든 팀원이 아닌 다른 팀원이 맡습니다(자기 검사는 관대해지니까).
4단계 · 검증
🤖 토큰값 전파
값 하나를 바꾸면 코드·사이트·설치기·문서에 빠짐없이 같이 고칩니다. 흩어진 곳을 찾아 직접 손대는 작업입니다.
토큰 값을 바꿀 때
🤖 토큰구조 설계검토
토큰을 새로 만들 때 짜임새·이름·다크모드가 의도에 맞는지 따져 고칠 점을 제안합니다.
토큰을 새로 설계할 때
🤖 레거시화면 빌드
옛 시안 화면을 지금 정본 부품으로 똑같이 다시 그립니다(피그마에 직접).
옛 화면을 재현할 때
🤖 라이브러리 빌드
피그마 컴포넌트(부품) 정의 자체를 새로 만들거나 변형세트로 묶습니다. 결과는 다른 팀원이 검사합니다.
피그마 부품을 빌드할 때

🔎 검사기 — 실수를 자동으로 막아주는 장치

비슷한 일을 하는 검사기끼리 6개 묶음으로 정리했습니다. 탭을 눌러 묶음별로 보세요. ("안 걸린다 = 잘 막고 있다"는 뜻이에요.)

컴포넌트를 만들 때 빠진 부품·잘못된 방식이 없는지

Gate 1 자동 감시
🔎 부품명세
컴포넌트 설명이 빠짐없이 적혀 있나
Gate 11 자동 감시
🔎 부품해부

Gate 12 자동 감시
🔎 아이콘 인스턴스
아이콘을 라이브러리 부품으로 제대로 넣었나
Gate 13 자동 감시
🔎 설치기 빌드검증
설치기 코드를 만든 사람 말고 다른 검증을 거쳤나

정해둔 값이 코드·사이트·설치기에 똑같이 나오는지

Gate 7 자동 감시
🔎 토큰값 일치
코드·사이트·설치기가 똑같은 값을 쓰나
Gate 9 자동 감시
🔎 사이즈·숫자 페이지
크기·숫자 값이 페이지에 맞게 나오나

필요한 토큰·변수·설치 파일이 빠짐없이 준비됐는지

Gate 6 자동 감시
🔎 설치기 누락
설치 파일에 토큰이 하나도 안 빠졌나
Gate 6b 자동 감시
🔎 설치기 최신

Gate 8 자동 감시
🔎 부품–변수 연결
컴포넌트가 쓰는 변수가 다 준비됐나

정해둔 표기·이름 규칙을 어긴 게 없는지

Gate 3 자동 감시
🔎 색상 규칙

Gate 15 자동 감시

Gate 10 자동 감시
🔎 문서 토큰이름

Gate 14 자동 감시
🔎 원본대조 문구

🔄 토큰편집 동기화기 늘 켜짐
토큰을 고치면 자동으로 다른 곳에도 맞춰줍니다.
⚙️ 피그마 색 차단 늘 켜짐
피그마에 색을 코드로 직접 쓰려 하면 막습니다 (반드시 변수로).
🔤 피그마 폰트 차단 늘 켜짐
